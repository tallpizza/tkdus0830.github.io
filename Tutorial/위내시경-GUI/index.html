    # Entry point

    디자인과 ai모델이 있는 상황에서 개발 시작

    개발환경 : Python

    # 구조

    >설명

    Controler 클래스에서 Login&out을 관리한다. 각각의 GUI 에선 정해진 로직을 처리하고 Controler 에 시그널을 보내는 형태이다.

    컨트롤러는 시그널에 따라 Login GUI 또는 Main GUI 를 보여준다.

    이유

    처음엔 로그인 성공시 Login GUI 위에 Main GUI 를 띄우는 형식이였는데, Login GUI 가 계속 백그라운드에서 떠있다보니 실행파일을 만들면 창이 2개가 뜨는 상황이 일어났다.

    # 메인화면

    설명

    ① 📷카메라 영상이 실시간으로 보여진다.

    • 크롭을 통해 원하는 부분만 선택해 보이게 할 수 있다.

    ② 여러가지 설정할 수 있는 🕹️버튼들이 있다.

    • AI 버튼 클릭시 내시경 영상을 ai 를 통해 분석한다. (결과 사진 저장)
    • REC 버튼 클릭시 영상을 저장한다.

    # 파이프를 통해 데이터 송수신

    >설명

    영상 저장과 AI분석은 프로세스를 나누었다. 다른 프로세스에 통신을 위해 파이프를 통해 데이터를 주고 받았다.

    이유

    한개의 프로세스로 실행하니 영상이 15프레임 정도까지 떨어져 성능개선을 위해 멀티 프로세스 사용 프로세스를 나눈후 녹화된 영상으로 실행했을때 50-60프레임정도가 나왔다.

    큐를 통해 데이터 전송을 하다가 아에 새로운 파일로 모듈들을 분리하자는 의견이 있어 파이프를 사용했다.

    소켓도 고려했지만 파이프가 전송속도가 더 빠를것 같아 파이프 사용

    # 레코드 모듈

    >설명

    레코드 모듈에선 파이프로 이미지를 받아서 저장한다. 멀티프로세스를 통해, 하나는 이미지를 받아서 큐에 저장하는 형태로, 다른 하나는 큐에서 이미지를 꺼내와서 파일을 만드는 형태이다.

    이유

    파이프가 양쪽이 동기화가 되어있는지 한쪽에서 받은 데이터를 읽지 않으면 다음을 전송을 안해서 큐를 추가적으로 사용했다.

    # AI 모듈

    설명

    레코드 모듈과 비슷한 형태이지만 다른점은 다시 메인 GUI 에 결과값을 보내줘야한다는 점이다. 똑같이 프로세스를 하나 더만들고 큐를 사용했다. 다만 AI결과를 위한 큐까지 총 AI 모듈에선 2개의 큐를 사용.

    이유

    레코드 모듈과 동일하다.
